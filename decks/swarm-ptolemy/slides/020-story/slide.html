Operating Model

One workspace, one theme, one proof loop

The workflow stays intentionally small so the agent has fewer places to drift and the validator has fewer blind spots.

1

Normalize the request

Put the goal, audience, tone, must-include points, and constraints into brief.md.

2

Build only in the allowed surfaces

Keep reusable design in theme.css and slide copy in slides/<NNN-id>/slide.html.

3

Preview and check the assembled deck

Runtime serves /decks/swarm-ptolemy/ directly, so the user previews the real assembled presentation instead of a hand-built artifact.

4

Finalize the proof package

finalize generates the PDF, screenshots, structured report, and summary once the rules pass cleanly.

Guardrails

The constraints are there to stop quiet drift.

Explicit workspace targeting means the agent cannot accidentally validate the wrong deck.
Asset boundaries keep slide content inside the current deck instead of reaching into sibling workspaces.
Unresolved brief markers block preview, check, export, and finalize before low-quality output leaks through.
Theme and slide rules keep the story editable without letting the agent bypass canvas or invent unsafe structure.
