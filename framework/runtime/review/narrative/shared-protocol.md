# Shared Narrative Review Protocol

## Scope
This is a narrative review of a rendered presentation artifact.

The reviewer sees only:
- PDF
- brief
- outline

The reviewer does not see:
- code
- source files
- package metadata
- runtime state
- implementation details

## What This Review Is For
This review checks whether the presentation communicates its message clearly and persuasively.

It may assess:
- thesis clarity
- audience fit
- section flow
- slide purpose
- claim framing
- evidence support
- objection handling
- ending or ask clarity

It must not assess:
- code quality
- technical implementation
- package structure
- runtime correctness
- non-narrative visual polish
- spacing, alignment, or styling unless they directly block understanding

## Review method
1. Read the brief once to understand intended audience and objective.
2. Scan the whole deck once for first impression.
3. Review every slide individually using the slide checklist below.
4. Review the deck as a whole using the deck-level checklist.
5. Write freeform feedback in the reviewer's own voice.

## Slide-by-slide checklist
For every slide, check:

### Role clarity
- Is it obvious why this slide exists?
- Does the slide have one clear job in the story?

### Message clarity
- Can the viewer say what this slide means after reading it?
- Is the intended takeaway actually clear?

### Copy quality
- Is the writing concise enough?
- Does any language feel vague, inflated, or confusing?
- Is jargon used only when the audience can support it?

### Claim and support
- If the slide makes a claim, is that claim supported?
- If the slide presents evidence, does it clearly connect to the point being made?

### Transition value
- Does the slide follow naturally from the previous one?
- Does it set up the next step in the argument?

### Audience fit
- Would the intended audience understand why this matters?
- Does the slide answer the question the audience is likely to ask at this point?

### Objection exposure
- Does the slide leave obvious skeptical gaps?
- Are there claims that feel unearned or insufficiently grounded?

## Deck-level checklist
After the slide pass, check:

- Is the core thesis clear?
- Does the deck build a coherent argument from start to finish?
- Does the sequence of slides feel intentional?
- Are the most important claims actually supported?
- Does the deck answer the likely audience objections?
- Is the ending, recommendation, or ask clear?

## Output
The reviewer should produce:
- unstructured freeform feedback
- narrative judgment only
- no code suggestions
- no file references
- no implementation talk
